Introduction

Pulse 2.0 is the internal work execution platform for DAASS, rebuilt from the ground up as a fully custom-coded application. This document covers the complete architecture, the reasoning behind key technical decisions, the development and deployment methodology, and the roadmap for expanding the platform to future DAASS projects.

This guide is intended for internal use by the operations team and any developer maintaining or extending the system. It assumes basic familiarity with the platform but no prior knowledge of the codebase.

Section 01

Project Overview

Pulse 2.0 is a role-based internal operations tool built for the DAASS team. It replaces the original AI-generated Lovable version, which suffered from data loss, limited customisability, and a fragmented admin experience. The new build provides persistent cloud-synced data, real-time team visibility, and a unified admin interface for EODs, payroll, and archive management.

Core Features

The platform is structured around five key areas:

Dashboard. Real-time task tracker, Loom recording link, quick-start task shortcuts, and the Start of Day (SOD) modal on every login.

Team. Live status board showing each team member's current activity (Working, Lunch, Offline) and active task, sourced from Supabase in real time.

EODs / Payroll. Admin-only view of all submitted End of Day reports, including tasks, hours, and Loom recording links.

Archive Files. Admin-only per-employee archive with EOD and Payroll sub-tabs.

Profile. Displays the user's SOD priorities and gratitude entry for the day (read-only).

Section 02

Lovable vs. Custom Build

The original Pulse was generated entirely through Lovable, an AI no-code builder. While fast to prototype, Lovable-generated applications have significant structural limitations that become blockers at production scale. Pulse 2.0 was rebuilt as a standard React/TypeScript application to resolve these constraints permanently.

FactorOriginal Pulse (Lovable)Pulse 2.0 (Custom)
Code OwnershipAI-generated; difficult to read or modify reliablyFull ownership; standard TypeScript anyone can maintain
Data PersistencelocalStorage only; data lost on browser clear or new deviceSupabase PostgreSQL; persistent, cloud-synced
CustomisationConstrained by Lovable's generation modelUnlimited; any feature can be built
Role-Based AccessBasic or absentAdmin / Employee routing with route-level protection
Real-Time FeaturesNoneSupabase Realtime subscription on team status
UI ConsistencyInconsistent; regeneration changes styling unpredictablyCustom design tokens, consistent across all pages
Tooling CostLovable subscription requiredZero tooling cost; all open-source
ScalabilityVery limited; adding features risks breaking existing codeStandard component architecture; fully scalable
DebuggingGenerated code is opaque; hard to trace bugsStandard React dev tools; TypeScript errors caught at build
DeploymentLovable-hosted; no control over environmentAny host: Vercel, Netlify, Railway, Cloudflare Pages
Key Decision

The switch from Lovable to a custom build was not a preference; it was a technical necessity. LocalStorage-only storage meant every team member effectively had a separate, private database. Admin visibility over EODs required persistent server-side storage.

Section 03

Technical Stack & Cost

Pulse 2.0 uses a modern, lightweight open-source stack. Every tool has a generous free tier and a large developer community, meaning support is readily available and there is no vendor lock-in.

Frontend

  • React 18. UI component library; industry standard for web applications.
  • TypeScript. Typed JavaScript; catches errors at build time before they reach users.
  • Vite 6. Fast build tool and development server.
  • React Router DOM v6. Client-side routing with role-based route protection.
  • Lucide React. SVG icon library; tree-shakeable, no performance cost.
  • Custom CSS variables. No UI framework dependency; keeps the bundle small and styling fully controlled.

Backend / Database (Supabase)

Supabase is an open-source Firebase alternative built on PostgreSQL. It provides a full database, real-time subscriptions, and a REST API out of the box. No backend server code is required.

Authentication & Costs

Authentication

  • Hardcoded USERS array in AuthContext; no external auth server required for current team size.
  • localStorage session persistence; survives page refresh, cleared on sign-out.
  • Session re-validated on load against USERS array; role changes take effect immediately without re-login.
  • Role-based routing: AdminRoute guard redirects employees away from admin pages at the router level.

Development Environment

  • Node.js 18+ with npm
  • Claude Code (AI-assisted development CLI)
  • VS Code with TypeScript and React extensions
  • .env file for environment variables (never committed to Git)

Development & Deployment

Pulse 2.0 is built and maintained using a three-stage pipeline: local development with Claude Code, version control via GitHub, and continuous deployment through a hosting platform. This process keeps all changes tracked, reversible, and live within minutes of being approved.

Step 1 — Local Development (Claude Code + VS Code)

All code is written and tested locally. Claude Code, Anthropic's AI development CLI, is used as the primary coding assistant, enabling rapid feature implementation with immediate TypeScript feedback.

  • Run npm run dev to launch the local development server at localhost:5173
  • All changes are tested in the browser before committing
  • Run npm run build to verify the TypeScript compilation passes before any push
  • Environment variables are stored in a local .env file (not committed to Git)

Step 2 — Version Control (GitHub)

All source code lives in a private GitHub repository. Every change is committed with a descriptive message, creating a complete audit trail of the project's evolution.

  • Repository: private, owned by DAASS organisation account
  • Branch strategy: main = production-ready code; feature branches for new work
  • The .env file is listed in .gitignore; credentials never enter the repository
  • Supabase SQL setup script (supabase_setup.sql) is committed for reproducibility

Step 3 — Deployment (Hosting Platform — TBA)

The hosting platform is connected to the GitHub repository. Every push to the main branch triggers an automatic rebuild; the live application updates within approximately 60–90 seconds.

  • Environment variables set in the hosting platform dashboard (not in code)
  • No server management required; the application is a static React SPA
  • HTTPS provided automatically by all candidate platforms
  • Custom domain can be connected at any point with a DNS record change

Future Roadmap & Scalability

Pulse 2.0 was architected with intentional simplicity so that it can grow without requiring a full rebuild. The stack — React, TypeScript, Vite, Supabase — is the same stack used by large production applications, meaning there is no ceiling on what can be built.

Near-Term (Next 1–3 Months)

  • GitHub repository setup and first production deployment to selected hosting platform
  • Supabase production environment configuration and table creation via supabase_setup.sql
  • Custom domain configuration (if applicable)
  • Team onboarding and password distribution

Medium-Term (3–6 Months)

  • Payroll calculation module; automated hour totals and pay summaries from EOD data (detailed in Section 06)
  • Email notifications; alert admins when an EOD is submitted
  • Admin analytics dashboard; weekly/monthly hours worked, task category breakdowns
  • Mobile-responsive layout; core pages optimised for phone and tablet use
  • Audit log; track who changed what and when

Platform Reuse for Future DAASS Applications

The Pulse 2.0 codebase is designed to be a reusable foundation. The design system, layout structure, authentication pattern, Supabase integration layer, and deployment pipeline are all generic enough to be adapted for other DAASS web applications with minimal configuration. Each new project would require:

  • A new Supabase project (free under the same account)
  • A new private GitHub repository (free)
  • A new hosting deployment (free on all candidate platforms)
  • Application-specific pages and database tables layered on top of the shared base
Platform Reuse Note

Any future DAASS web application — internal tools, client portals, academy platforms — can be bootstrapped from the Pulse 2.0 codebase within hours rather than weeks. The foundation is already built, tested, and deployed.

Section 06

EOD Reporting & Automated Payroll

Pulse already does the hard part: tracking what everyone worked on and for how long. This section closes the loop by saving that data properly, rolling it up into weekly payroll totals, and giving admins a clean one-click export. No more manual math, no more digging through EOD emails.

What Pulse Already Does Well

Task-level time tracking. Employees log tasks with start/stop precision. Subtasks, notes, and links are all captured.

Session vs. task time distinction. Pulse correctly separates total session time from actual task time, and we pay for task time only.

EOD modal. The Submit & End Day flow already surfaces the right data: tasks completed, durations, open items, Loom links, and notes.

Admin visibility. Live status views and per-user history exist. The data is there.

The Gap This Section Closes

IssueImpact
EOD data doesn't persistThe modal collects everything correctly but doesn't save daily task totals in a structured, payroll-ready format.
No weekly rollupThere is no accumulation of daily totals across the work week. Calculating weekly hours requires manual effort.
EOD emails are unthreadedAdmin email reports are sent as individual messages. A single ongoing thread per employee is needed.
Payroll is manualWithout structured totals, someone has to manually calculate hours and cross-reference rates.

Section 06 · Phase 01

EOD Reporting, Fixed

The EOD modal already shows the right data. Phase 01 makes it count by persisting that data, threading the admin email, and structuring every send for fast review.

Save to Database on Submit

When an employee clicks Submit & End Day, Pulse saves a structured daily record: employee, date, total billable task time, individual task breakdown, open items, notes, and Loom link. This record becomes the single source of truth for that employee's day.

Threaded EOD Emails

Each employee has one persistent email thread. Every EOD report is sent as a reply to that thread, so the admin inbox shows a single ongoing conversation per person — Monday, Tuesday, Wednesday building on each other like a log. Technically, this works by storing each employee's thread ID after the first email and using email reply headers (In-Reply-To) for every subsequent send.

Structured Email Format

Each threaded EOD email includes:

  • Date and total billable task time for the day
  • Task-by-task breakdown with durations
  • What's still open
  • Notes and Loom link if provided
Phase 01 Outcome

Phase 01 alone delivers immediate value. Threaded emails replace inbox clutter, and persistent records become available for any future analysis, even before payroll logic is wired up in later phases.

Section 06 · Phase 02

Daily Task-Time Persistence

Phase 01 saves the EOD record. Phase 02 makes those records queryable for payroll, with per-employee work-week awareness so the rollup respects each person's actual schedule.

Daily Totals Stored Per Employee

Each saved EOD record includes a clean billable_minutes field — the sum of all task durations for that day, excluding session time, lunch, and idle time. This is the number payroll is calculated from.

Configurable Work Week Per Employee

Not everyone works the same days. Each employee's profile includes a work-week configuration — for example, one team member works Sunday through Thursday instead of Monday through Friday. The weekly rollup respects each person's schedule individually.

Section 06 · Phase 03

Weekly Payroll Rollup

At the end of each work week, Pulse calculates each employee's total billable hours and what they're owed. The rollup runs automatically against the persisted daily totals from Phase 02.

Weekly Accumulation

Daily billable totals accumulate automatically across the configured work week. No manual addition. The weekly total is always current and visible to admins.

Per-Employee Hourly Rates

Each employee has a private hourly rate stored in admin settings. Rates are never visible to the employee, only to admins. Every minute is counted: 3 hours 31 minutes is paid as 3.52 hours, not rounded down.

Calculated Weekly Pay

Weekly hours × hourly rate = amount owed. Calculated automatically, visible in the admin panel.

Privacy Principle

Hourly rates live exclusively in admin-only settings. No employee can access, infer, or view another person's rate or pay. Phase 03 introduces the rate field but inherits the existing role-based access controls established in Section 01.

Section 06 · Phase 04

One-Click Payroll Export

A single button in the Pulse admin panel exports the current week's payroll summary as a spreadsheet, opening directly in Google Sheets or Excel with no reformatting needed.

Export Contents

ColumnDescription
Employee nameFull name as stored in profile
Week ofDate range covering the configured work week
Days workedCount of days with submitted EOD records
Total billable hoursPrecise decimal hours, not rounded
Hourly rateFrom admin settings (admin view only)
Amount owedHours × rate (admin view only)

Access & Privacy

  • Rates are invisible to employees. Hourly rates live in admin-only settings.
  • Export is admin-gated. The payroll export button only appears for admin-role users.
  • EOD data is per-user. Employees can only see their own history.
  • No new surveillance. This adds payroll logic on top of existing task data; no screenshots, idle detection, or monitoring beyond what Pulse already captures.

Conclusion

Pulse 2.0 represents a meaningful step forward in how the DAASS team tracks, coordinates, and documents its work. By moving from a Lovable-generated prototype to a purpose-built application, the team now has a platform it fully owns — one that can be changed, extended, and scaled without constraints. The database is persistent, the UI is consistent, and the admin workflow is significantly faster than before.

The methodology established here — Claude Code for development, GitHub for version control, and a CI/CD-connected hosting platform — is repeatable for every future DAASS web project. The tooling cost is effectively zero, and the process from idea to deployed feature is measured in hours, not weeks. Section 06 demonstrates how new operational capability can be layered on top of the existing foundation without rebuilding any of it.

Key Takeaways

  1. The full codebase is owned by DAASS and can be modified or extended freely at any time.
  2. Supabase ensures all data is persistent and visible across every device and session.
  3. The GitHub → hosting pipeline means updates are live within 90 seconds of approval.
  4. The same stack can serve as the foundation for every future DAASS web application.
  5. Current monthly operating cost is $0 on free tiers across all services.
  6. The EOD & Payroll workflow specified in Section 06 closes the operational loop with no new tooling cost.
Final Note

The best internal tool is one the team actually uses. Pulse 2.0 was built around how the team already works, not the other way around.
